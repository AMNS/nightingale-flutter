NIGHTINGALE RUST PORTING ROADMAP

EXECUTIVE SUMMARY
=================

This roadmap charts a systematic port of Nightingale's music engraving engine
to Rust, organized by dependency layers. The strategy is:

1. Port LAYERS FIRST (not files), in dependency order
2. Build each layer with tests validated against real .ngl files
3. Replace QuickDraw/Carbon APIs with a rendering trait layer
4. Maintain file format compatibility (N103/N105 → N106)

TOTAL SOURCE TO PORT: ~32,000 lines of C++ across 22 core files
ESTIMATED COMPLEXITY: High (engraving algorithms are intricate)
ESTIMATED DURATION: 6-9 months for complete port + testing


LAYER BREAKDOWN
===============

LAYER 0: FOUNDATIONAL TYPES (Headers, no code) - ~2,800 lines
  Files: 7 header files in Precomps/
  Work: Define Rust structs/enums/constants
  Deliverable: types.rs, constants.rs, enums.rs, objects.rs
  Tests: File format round-trip tests (load .ngl → validate against Python converter)
  Duration: 2-3 weeks (meticulous work, many L/G/P domain annotations to port)

LAYER 1: INFRASTRUCTURE (Allocator, strings, I/O) - ~3,000 lines
  Files: StringPool.cp, Objects.cp, HeapFileIO.cp
  Work: Object allocator, string pool, binary file I/O
  Deliverable: heap.rs, allocator.rs, string_pool.rs, file_io.rs
  Tests: Load/save round-trip on real .ngl files (N103, N105)
  Duration: 3-4 weeks (straightforward translation; test against real files)

LAYER 2: MUSICAL CONTEXT (~1,200 lines)
  Files: Context.cp
  Work: Context propagation (clef, key sig, time sig, dynamics)
  Deliverable: context.rs
  Tests: Verify context state after each layout step
  Duration: 1-2 weeks (moderate complexity; heavily used downstream)

LAYER 3: SPACING AND TIMING (~1,981 + 1,781 lines)
  Files: SpaceTime.cp, SpaceHighLevel.cp
  Work: Horizontal/temporal spacing, duration conversions, justification
  Deliverable: spacing.rs, layout.rs
  Tests: Compare spacing output to original on test scores
  Duration: 4-6 weeks (complex logic; high-value to get right)

LAYER 4: UTILITY FUNCTIONS (~1,695 lines)
  Files: Utility.cp
  Work: Stem positioning, coordinate conversions, geometry helpers
  Deliverable: utility.rs
  Tests: Unit tests against known values; compare stem positions
  Duration: 2-3 weeks (mostly independent functions)

LAYER 5: ENGRAVING ALGORITHMS (~7,000+ lines)
  Files: Beam.cp, GRBeam.cp, Slurs.cp, Tuplet.cp
  Work: Core engraving (beam angles, Bezier curves, tuplet logic)
  Deliverable: beam.rs, grace_beam.rs, slurs.rs, tuplets.rs
  Tests: Visual regression tests (compare output before/after)
  Duration: 6-8 weeks (most complex; heavily mathematical)

LAYER 6: RENDERING ABSTRACTION (New layer - ~2,000 lines expected)
  Files: Create new Rust trait layer (no C++ source to port)
  Work: Define abstract "draw instruction" format
  Deliverable: rendering.rs, draw_instructions.rs
  Tests: Generate draw instructions; verify instruction sequence
  Duration: 2-3 weeks (design work; proof-of-concept implementation)

LAYER 7: DRAWING / OUTPUT (Port drawing logic, discard QuickDraw) - ~9,500 lines
  Files: DrawHighLevel.cp, DrawObject.cp, DrawNRGR.cp, PS_Stdio.cp
  Work: Convert to emit rendering instructions (not QuickDraw calls)
  Deliverable: draw.rs, draw_objects.rs, rendering_backend.rs
  Tests: Snapshot tests (render to SVG or image, compare)
  Duration: 4-6 weeks (straightforward translation once trait is designed)

CRITICAL SUCCESS FACTORS
========================

1. FILE FORMAT FIDELITY
   - Load N103/N105 .ngl files without data loss
   - Be able to write N106 format (can be new)
   - Validate via Python NGL→MusicXML converter (existing reference)

2. MATHEMATICAL ACCURACY
   - Beam angles, spacing, stem positions must MATCH C++ output
   - Use golden reference files (real scores from Nightingale 6.0)
   - Side-by-side visual comparison on test suite

3. ARCHITECTURAL DECOUPLING
   - Separate data model from rendering via trait abstraction
   - Do NOT hardcode QuickDraw/Carbon—use render instructions
   - Enable future: Flutter UI, PDF output, MusicXML export

4. REGRESSION TESTING
   - Snapshot tests for each engraving subsystem
   - Load real .ngl files; verify output matches reference
   - Automated CI/CD on each commit


TESTING STRATEGY
================

UNIT TESTS (each layer, as ported):
  - Coordinate conversion macros
  - Spacing calculations
  - Beam angle formulas
  - Bezier curve points
  - Tuplet duration calculations

INTEGRATION TESTS (each phase):
  - Load real .ngl file
  - Verify object counts, structure
  - Compare layout output (spacing, beam angles, etc.)

GOLDEN FILE TESTS:
  - Use reference .ngl files from Nightingale 6.0
  - Load in Rust → render → compare to saved "golden" output
  - Fail if output differs (catch regressions immediately)

VISUAL TESTS:
  - Render to SVG/PNG
  - Human inspection (UI team)
  - Automated screenshot diff (perceptual hashing)


RISKS & MITIGATIONS
====================

RISK: Coordinate system confusion (DDIST, STDIST, staff vs. paper coords)
MITIGATION: Create comprehensive coordinate conversion tests. Document DDIST
            semantics in comments (1/16 point resolution, ±2048 point range).

RISK: Macro-heavy C++ code doesn't port cleanly to Rust
MITIGATION: Convert macros to inline functions. Build a macros.rs module to
            centralize conversions.

RISK: Engraving algorithms are intricate; easy to introduce bugs
MITIGATION: Golden file tests. Compare output pixel-by-pixel on reference scores.
            Run alongside C++ implementation in test mode.

RISK: File format compatibility issues (bitfield packing, endianness)
MITIGATION: Use serde with careful serialize/deserialize implementations.
            Test N105 files; verify conversion to modern format.

RISK: Performance regression (Rust is fast, but VM overhead may hurt)
MITIGATION: Profile on large scores (100+ measures). Use criterion.rs for
            benchmark suite. Aim for <2x C++ speed (likely faster).
