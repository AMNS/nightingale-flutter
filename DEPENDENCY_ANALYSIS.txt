NIGHTINGALE C++ SOURCE DEPENDENCY ANALYSIS
==========================================

This analysis maps the dependency chain of files destined for Rust porting.
Dependency notation: X depends on Y means X #includes Y or uses types from Y.

LAYER 0: BASE TYPE DEFINITIONS (No Dependencies Within This Set)
=================================================================

1. Precomps/NBasicTypes.h (206 lines)
   - Exports: DDIST, STDIST, SHORTQD, DPoint, DRect, KSINFO, TSINFO, LINK, HEAP
   - Coordinate types, enumerations for object types (HEADERtype, SYNCtype, etc.)
   - Key constants: STD_LINEHT=8, coordinate type ranges
   - No dependencies within target set

2. Precomps/NLimits.h (77 lines)
   - Exports: MAXSTAVES, MAXVOICES, MAXCHORD, MAX_L_DUR, MAX_SCOREFONTS, limits
   - Constant definitions only, no code
   - No dependencies within target set

3. Precomps/defs.h (601 lines)
   - Exports: Enums (l_dur codes, accidentals, output device types, etc.)
   - Music notation constants (MCH_trebleclef, MCH_sharp, etc.)
   - Macros: ABS, MIN, MAX, VISIBLE, etc.
   - Coordinate conversion macros (std2d, d2std, pt2d, d2p, etc.)
   - No dependencies within target set

4. Precomps/style.h (137 lines)
   - Exports: Notation style macros (GRACESIZE, SMALLSIZE, HeadWidth, FlagLeading)
   - Accidental, note, measure, tuplet, ending style parameters
   - Depends on: defs.h (uses STD_LINEHT macro), NBasicTypes.h (DDIST/STDIST)

5. Precomps/NObjTypes.h (994 lines)
   - Exports: All major object/subobject structs (HEADER, TAIL, SYNC, NOTE, ANOTE,
     MEASURE, STAFF, CLEF, KEYSIG, TIMESIG, BEAMSET, SLUR, TUPLET, etc.)
   - L/G/P domain annotations on fields
   - Complex nested structures with bitfields removed for compatibility
   - Depends on: NBasicTypes.h (LINK, DDIST, SHORTQD, OBJECTHEADER macro uses these)

6. Precomps/NObjTypesN105.h (642 lines)
   - Exports: N105 file-format variants of all structs (bitfield versions)
   - Used for reading legacy files, converts to modern structures
   - Depends on: NBasicTypes.h (same types)

7. Precomps/NDocAndCnfgTypes.h (785 lines)
   - Exports: DOCUMENTHDR, SCOREHEADER, Document struct, Configuration, etc.
   - File format version constants, font records, MIDI preferences
   - Depends on: NBasicTypes.h, NObjTypes.h (uses LINK, OBJECTHEADER)

PORTING ORDER (Dependency-First)
=================================

PHASE 1: Core Types (Headers only → Rust structs)
  1. NBasicTypes.h → types.rs (DDIST, LINK, etc.)
  2. NLimits.h → constants.rs
  3. defs.h → enums.rs, macros.rs
  4. style.h → style.rs
  5. NObjTypes.h → objects.rs (all object structs)
  6. NObjTypesN105.h → objects_n105.rs (legacy format converters)
  7. NDocAndCnfgTypes.h → document.rs

PHASE 2: Infrastructure
  8. StringPool.cp → string_pool.rs
  9. Objects.cp → heap.rs, allocator.rs
  10. HeapFileIO.cp → file_io.rs

PHASE 3: Musical/Layout Logic
  11. Context.cp → context.rs
  12. SpaceTime.cp → spacing.rs
  13. SpaceHighLevel.cp → layout.rs
  14. Utility.cp → utility.rs

PHASE 4: Engraving Algorithms
  15. Beam.cp → beam.rs
  16. GRBeam.cp → grace_beam.rs
  17. Slurs.cp → slurs.rs
  18. Tuplet.cp → tuplets.rs

PHASE 5: Rendering (Drawing)
  → Drawing code will map to a Rust rendering trait (not QuickDraw/Carbon APIs)


KEY EXPORTS BY PHASE
====================

PHASE 1 Types:
  - All struct definitions (ANOTE, MEASURE, SYNC, BEAMSET, SLUR, TUPLET, etc.)
  - Coordinate types and conversions
  - Constants and enumerations
  - Macros (convert to functions/constants in Rust)

PHASE 2 Infrastructure:
  - HeapAlloc, LinkToPtr, GetPObject family functions
  - DuplicateObject, InitObject
  - String pool reference management
  - Linked-list traversal primitives

PHASE 3 Musical Logic:
  - CalcYStem, GetNoteYStem - note vertical position in staff
  - GetContext, ContextKeySig, ContextTimeSig - musical context propagation
  - CalcSpaceNeeded, FillSpaceMap - spacing computations
  - JustifySystems, RespaceBars - system-level layout

PHASE 4 Engraving:
  - DoBeam, CreateBeams - beam creation and stem angle calculation
  - AnalyzeBeamset, BuildBeamDrawTable - beam rendering data
  - GetSlurPoints - Bezier curve control points for slurs/ties
  - CreateTUPLET, SetTupletYPos - tuplet bracket positioning

PHASE 5 Rendering:
  - DrawBEAMSET, DrawSLUR, DrawTUPLET - delegate to Flutter
  - Note/rest drawing decomposes to: (glyph_id, x, y, size)
  - Ledger lines: (line_y, x_left, x_right) instructions


NOTES FOR RUST IMPLEMENTATION
=============================

1. LINK is currently unsigned short (0..n objects in heap).
   In Rust: could be u16 for memory compat, or lifetime-tracked references.
   Decision: probably remain u16 for file format compatibility.

2. Many functions have Mac OS type signatures (Rect, Point, Handle, etc).
   These will disappear in Rust—use simple (x, y, width, height) tuples.

3. Macros like std2d(), d2std(), etc. become Rust inline functions.
   These are heavily used in layout code; make them inlined.

4. HeapAlloc/LinkToPtr are core to the current design.
   Rust will use a custom allocator or Vec<Object> under the hood.
   The LINK→Object mapping is abstracted via traits.

5. PostScript primitives (PS_Char, PS_Line) will be replaced by a
   rendering trait that outputs drawing instructions for Flutter.

6. File format round-trip testing is critical:
   - Load .ngl (N103/N105) → Rust data structures
   - Verify musical content via existing Python NGL→MusicXML
   - Export MusicXML, re-import, verify equivalence
   This should be Phase 1 acceptance test.
