================================================================================
NIGHTINGALE C++ DRAWING PIPELINE - EXECUTIVE SUMMARY
================================================================================

This document summarizes the key architecture of the Nightingale drawing
pipeline for porting to Rust + Flutter.

CORE CONCEPT
============

The drawing pipeline is a RECURSIVE DESCENT through the object hierarchy with
CONTEXT-BASED COORDINATE TRANSFORMATION:

    PAGE → SYSTEM → STAFF → MEASURE → SYNC → NOTE

At each level, CONTEXT is updated with absolute page-relative DDIST coordinates.
At draw time, DDIST values are converted to pixels (d2p macro) which varies by
magnification level.

KEY PRINCIPLE: Data model uses DDIST (1/16 point). Rendering converts DDIST to
pixels at draw time, not at model-build time.

COORDINATE SYSTEM
=================

DDIST (signed short, range ±2048 points, resolution 1/16 point):
    - Used in data model (score positions)
    - Used in CONTEXT structure (positions, dimensions)
    - Used in PostScript output directly

Pixels (int):
    - Used in QuickDraw calls for screen rendering
    - Computed from DDIST via d2p() macro
    - Depends on magnification level (nonlinear for odd magnifications)

Conversion:
    pixel = paper_origin + d2p(ddist)
    
where d2p(ddist) uses bit-shifting based on magnification:
    - 100% magnify: simple shift by magShift
    - 150% magnify: shift and multiply by 1.5 (uses magRound for rounding)

CONTEXT STRUCTURE (Critical!)
=============================

Passed to ALL drawing functions. Tracks:
    - Position: systemTop, systemLeft, staffTop, staffLeft, staffRight,
               measureTop, measureLeft (all DDIST, page-relative)
    - Visibility: visible, staffVisible, measureVisible, inMeasure
    - Layout: staffHeight, staffLines, staffHalfHeight, showLines, showLedgers
    - State: clefType, keySignature, timeSignature, fontSize
    - Paper: paper (Rect in window coords), sheetNum, systemNum

Helper macro: LNSPACE(pCont) = staffHeight / (staffLines - 1) = space between
              staff lines in DDIST

DRAWING FUNCTIONS HIERARCHY
============================

HIGH-LEVEL DISPATCH:

    DrawRange()              [DrawHighLevel.cp:709]
        ├─ SetMusicPort()    - Sets font, magnification
        ├─ DrawMasterRange() - Master page
        ├─ DrawFormatRange() - Format page
        └─ DrawScoreRange()  - Normal score
            └─ HiliteScoreRange() - Selection hiliting

SCORE DRAWING DISPATCH (DrawScoreRange walks object list):

    DrawPAGE()          - Sets paper rect, calls GetAllContexts
    DrawSYSTEM()        - Checks visibility, frames for debug
    DrawSTAFF()         - Iterates substaves, updates context
    DrawCONNECT()       - Staff brackets/braces
    DrawMEASURE()       - Barlines, measure numbers
    DrawSYNC()          - Notes & rests
        ├─ DrawNote()   - Single note: head, stem, flags, accidental, dots, ledgers
        │   ├─ DrawAcc()         - Accidental symbol
        │   ├─ DrawNCLedgers()   - Ledger lines
        │   ├─ DrawNotehead()    - Notehead glyph
        │   ├─ DrawAugDots()     - Rhythmic dots
        │   └─ [Stem/Flags]      - Via MoveTo/Line
        └─ DrawRest()   - Rest symbol
    DrawBEAMSET()       - Beam lines
    DrawTUPLET()        - Tuplet brackets
    DrawSLUR()          - Slur curves
    ... (other object types)

CRITICAL DRAWING PATTERNS
==========================

1. CONTEXT UPDATE (DrawSTAFF line 614-623):
    Context values are ACCUMULATED as we descend the tree:
    
    pContext->staffTop = aStaff->staffTop + pContext->systemTop;
    pContext->staffLeft = aStaff->staffLeft + pContext->systemLeft;
    
    This makes CONTEXT stateful—must be a mutable reference in Rust.

2. COORDINATE TRANSFORMATION (DrawNote line 769-773):
    
    xhead = pContext->paper.left + d2p(xd);
    yhead = pContext->paper.top + d2p(yd);
    
    This happens at DRAW TIME, after all layout computation.

3. OBJECT DISPATCHING (DrawScoreRange lines 366-500):
    
    for (pL = fromL; pL != toL; pL = RightLINK(pL)) {
        switch (ObjLType(pL)) {
            case PAGEtype: DrawPAGE(...); break;
            case STAFFtype: DrawSTAFF(...); break;
            ...
        }
    }
    
    This is a LINKED-LIST traversal, not a tree traversal.

4. VISIBILITY OPTIMIZATION (DrawScoreRange):
    
    if (drawAll || !LinkVALID(pL) || SectRect(&bBox, &updateRect, &result)) {
        DrawXXX(...);
    }
    
    Skips drawing if bounding box doesn't intersect update region.

5. OUTPUT DEVICE BRANCHING (DrawNote line 747):
    
    switch (outputTo) {
        case toScreen:
        case toBitmapPrint:
        case toPICT:
            // Use QuickDraw (MoveTo, LineTo, etc.)
            // Convert to pixels via d2p()
            break;
        case toPostScript:
            // Use PS_* functions
            // Use DDIST directly, no d2p()
            break;
    }
    
    CRITICAL: Two parallel code paths. Rust must preserve this abstraction.

VOICE & DIMMING
===============

Voice-based coloring:
    ForeColor(Voice2Color(doc, aNote->voice));

Dimming for non-active voices:
    dim = (outputTo == toScreen && !LOOKING_AT(doc, aNote->voice));
    if (dim) PenPat(NGetQDGlobalsGray());

RENDERING ABSTRACTION REQUIRED FOR RUST/FLUTTER
=================================================

Instead of direct QuickDraw calls, Rust must generate a list of drawing
commands:

    struct DrawingCommand {
        command_type: CommandType,  // MoveTo, LineTo, DrawGlyph, etc.
        x: i32,
        y: i32,
        glyph: Option<u8>,
        color: Option<RGB>,
        dim: bool,
    }

OR a trait-based approach:

    trait RenderingBackend {
        fn move_to(&mut self, x: f32, y: f32);
        fn line_to(&mut self, x: f32, y: f32);
        fn draw_glyph(&mut self, glyph: u8, appearance: u8, dim: bool);
        fn set_color(&mut self, color: RGB);
        fn set_line_width(&mut self, width: f32);
    }

Flutter then consumes this output and renders via CustomPainter or similar.

IMPORTANT NOTES FOR PORTING
=============================

1. MAGNIFICATION IS CRITICAL
   - d2p() varies nonlinearly with magnification
   - Must preserve exact rounding behavior to avoid beat/stem misalignment
   - Global variables: magMagnify, magShift, magRound

2. FONT METRICS ARE PLATFORM-SPECIFIC
   - MusCharXOffset(), MusCharYOffset() provide per-font fine-tuning
   - Must be adapted for SMuFL fonts
   - Sonata font assumptions may not hold

3. CHORDS ARE COMPLEX
   - Notes in chords may share stems
   - Accidental/dot positioning depends on chord context
   - ChordNoteToLeft(), ChordNoteToRight() determine stem side

4. GRACE NOTES ARE SEPARATE
   - GRSYNC ≠ SYNC
   - DrawGRNote() is similar to DrawNote() but simpler
   - Grace note scaling is separate from main note sizing

5. SELECTION STATE IS DEFERRED
   - DrawScoreRange() draws opaque
   - HiliteScoreRange() redraws selected objects with hiliting
   - Allows clean UI separation

6. LINKED-LIST TRAVERSAL, NOT TREE
   - Objects are stored in linked lists: head → ... → tail
   - RightLINK(pL) traverses right, LeftLINK(pL) traverses left
   - LSSearch() finds objects by type; SSearch() for subsequences

7. MEASURE VISIBILITY OPTIMIZATION
   - drawAll boolean reduces work
   - Only draw spanning objects if measure not visible
   - Helps performance on large scores
